We\textquotesingle{}d love you to help out with mockery and no contribution is too small.\hypertarget{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1138}{}\doxysection{\texorpdfstring{Reporting Bugs}{Reporting Bugs}}\label{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1138}
Issues can be reported on the \href{https://github.com/padraic/mockery/issues}{\texttt{ issue tracker}}. Please try and report any bugs with a minimal reproducible example, it will make things easier for other contributors and your problems will hopefully be resolved quickly.\hypertarget{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1139}{}\doxysection{\texorpdfstring{Requesting Features}{Requesting Features}}\label{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1139}
We\textquotesingle{}re always interested to hear about your ideas and you can request features by creating a ticket in the \href{https://github.com/padraic/mockery/issues}{\texttt{ issue tracker}}. We can\textquotesingle{}t always guarantee someone will jump on it straight away, but putting it out there to see if anyone else is interested is a good idea.

Likewise, if a feature you would like is already listed in the issue tracker, add a \+:+1\+: so that other contributors know it\textquotesingle{}s a feature that would help others.\hypertarget{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1140}{}\doxysection{\texorpdfstring{Contributing code and documentation}{Contributing code and documentation}}\label{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1140}
We loosely follow the \href{https://github.com/php-fig/fig-standards/blob/master/accepted/PSR-1-basic-coding-standard.md}{\texttt{ PSR-\/1}} and \href{https://github.com/php-fig/fig-standards/blob/master/accepted/PSR-2-coding-style-guide.md}{\texttt{ PSR-\/2}} coding standards, but we\textquotesingle{}ll probably merge any code that looks close enough.


\begin{DoxyItemize}
\item Fork the \href{https://github.com/padraic/mockery}{\texttt{ repository}} on Git\+Hub
\item Add the code for your feature or bug
\item Add some tests for your feature or bug
\item Optionally, but preferably, write some documentation
\item Optionally, update the CHANGELOG.\+md file with your feature or \href{http://en.wikipedia.org/wiki/Backward_compatibility}{\texttt{ BC}} break
\item Send a \href{https://help.github.com/articles/creating-a-pull-request}{\texttt{ Pull Request}} to the correct target branch (see below)
\end{DoxyItemize}

If you have a big change or would like to discuss something, create an issue in the \href{https://github.com/padraic/mockery/issues}{\texttt{ issue tracker}} or jump in to \#mockery on freenode

\doxylink{namespace_mockery}{Mockery} may have several active branches at any one time and roughly follows a \href{https://igor.io/2013/10/21/git-branching-model.html}{\texttt{ Git Branching Model}}. Generally, if you\textquotesingle{}re developing a new feature, you want to be targeting the master branch, if it\textquotesingle{}s a bug fix, you want to be targeting a release branch, e.\+g. 0.\+8.\hypertarget{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1142}{}\doxysection{\texorpdfstring{Testing Mockery}{Testing Mockery}}\label{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1142}
To run the unit tests for \doxylink{namespace_mockery}{Mockery}, clone the git repository, download \doxylink{namespace_composer}{Composer} using the instructions at \href{http://getcomposer.org/download/}{\texttt{ http\+://getcomposer.\+org/download/}}, then install the dependencies with {\ttfamily php /path/to/composer.phar install}.

This will install the required dev dependencies and create the autoload files required by the unit tests. You may run the {\ttfamily vendor/bin/phpunit} command to run the unit tests. If everything goes to plan, there will be no failed tests!\hypertarget{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1143}{}\doxysection{\texorpdfstring{Debugging Mockery}{Debugging Mockery}}\label{md_vendor_2mockery_2mockery_2_c_o_n_t_r_i_b_u_t_i_n_g_autotoc_md1143}
\doxylink{namespace_mockery}{Mockery} and its code generation can be difficult to debug. A good start is to use the {\ttfamily Require\+Loader}, which will dump the code generated by mockery to a file before requiring it, rather than using eval. This will help with stack traces, and you will be able to open the mock class in your editor.
