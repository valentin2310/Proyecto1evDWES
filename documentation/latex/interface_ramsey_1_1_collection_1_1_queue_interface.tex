\doxysubsection{Detailed Description}
A queue is a collection in which the entities in the collection are kept in order.

The principal operations on the queue are the addition of entities to the end (tail), also known as {\itshape enqueue}, and removal of entities from the front (head), also known as {\itshape dequeue}. This makes the queue a first-\/in-\/first-\/out (FIFO) data structure.

Besides basic array operations, queues provide additional insertion, extraction, and inspection operations. Each of these methods exists in two forms\+: one throws an exception if the operation fails, the other returns a special value (either {\ttfamily null} or {\ttfamily false}, depending on the operation). The latter form of the insert operation is designed specifically for use with capacity-\/restricted {\ttfamily Queue\+Interface} implementations; in most implementations, insert operations cannot fail.

Queues typically, but do not necessarily, order elements in a FIFO (first-\/in-\/first-\/out) manner. Among the exceptions are priority queues, which order elements according to a supplied comparator, or the elements\textquotesingle{} natural ordering, and LIFO queues (or stacks) which order the elements LIFO (last-\/in-\/first-\/out). Whatever the ordering used, the head of the queue is that element which would be removed by a call to remove() or poll(). In a FIFO queue, all new elements are inserted at the tail of the queue. Other kinds of queues may use different placement rules. Every {\ttfamily Queue\+Interface} implementation must specify its ordering properties.

The {\ttfamily offer()} method inserts an element if possible, otherwise returning {\ttfamily false}. This differs from the {\ttfamily add()} method, which can fail to add an element only by throwing an unchecked exception. The {\ttfamily offer()} method is designed for use when failure is a normal, rather than exceptional occurrence, for example, in fixed-\/capacity (or "{}bounded"{}) queues.

The {\ttfamily remove()} and {\ttfamily poll()} methods remove and return the head of the queue. Exactly which element is removed from the queue is a function of the queue\textquotesingle{}s ordering policy, which differs from implementation to implementation. The {\ttfamily remove()} and {\ttfamily poll()} methods differ only in their behavior when the queue is empty\+: the {\ttfamily remove()} method throws an exception, while the {\ttfamily poll()} method returns {\ttfamily null}.

The {\ttfamily element()} and {\ttfamily peek()} methods return, but do not remove, the head of the queue.

{\ttfamily Queue\+Interface} implementations generally do not allow insertion of {\ttfamily null} elements, although some implementations do not prohibit insertion of {\ttfamily null}. Even in the implementations that permit it, {\ttfamily null} should not be inserted into a queue, as {\ttfamily null} is also used as a special return value by the {\ttfamily poll()} method to indicate that the queue contains no elements.

{\footnotesize\ttfamily add (\begin{DoxyParamCaption}\item[{mixed}]{\$element }\end{DoxyParamCaption})}

Ensures that this queue contains the specified element (optional operation).

Returns {\ttfamily true} if this queue changed as a result of the call. (Returns {\ttfamily false} if this queue does not permit duplicates and already contains the specified element.)

Queues that support this operation may place limitations on what elements may be added to this queue. In particular, some queues will refuse to add {\ttfamily null} elements, and others will impose restrictions on the type of elements that may be added. Queue classes should clearly specify in their documentation any restrictions on what elements may be added.

If a queue refuses to add a particular element for any reason other than that it already contains the element, it must throw an exception (rather than returning {\ttfamily false}). This preserves the invariant that a queue always contains the specified element after this call returns.
